## Threshold analysis

The ÔÇť4 files / 5 acceptance criteriaÔÇŁ rule is directionally reasonableÔÇöbecause multi-file scope correlates with lower successÔÇöbut it is not calibrated to the sharpest empirical signal observed in public benchmarks, and it misses several high-leverage predictors.

On file count: the SWE-PolyBench authors explicitly treat number of modified files as a proxy for modification complexity, then show that file count alone lacks detail, and still report a marked performance decrease once tasks require edits across three or more files (dropping below ~10% in their experiments). ţłÇciteţłéturn5view1ţłéturn5view0ţłü The SWE-bench Verified failure analysis likewise states that performance declines as complexity increases, ÔÇťespecially for multi-file modification tasks.ÔÇŁ ţłÇciteţłéturn4view0ţłéturn4view1ţłü This suggests that **a 4-file trigger is likely too permissive for ÔÇťcapable-but-limitedÔÇŁ agents**, at least if your internal tasks resemble benchmark-style repo issues in that multi-file implies cross-module coordination. However, it also supports your critique: ÔÇťfiles changedÔÇŁ is substrate-blind, and some multi-file tasks will be easy (e.g., adjusting multiple config files) while some single-file tasks are hard (deep root-cause localization, sensitive invariants). ţłÇciteţłéturn5view1ţłéturn4view2ţłü

On acceptance criteria: there is not a strong empirical basis in SWE-bench-like benchmarks for ÔÇť5 acceptance criteriaÔÇŁ as an inflection point, because those benchmarks do not standardize AC structure the way your internal tickets do. ţłÇciteţłéturn3view2ţłéturn4view1ţłü The best you can say from the evidence base is that multi-objective work and constraint interactions plausibly increase failure probability, and that failures can include ÔÇťcontext amnesiaÔÇŁ and late-stage loops where the agent loses track of original objectivesÔÇöpatterns that would be exacerbated by many criteria and constraints. ţłÇciteţłéturn4view1ţłü But treating ÔÇť5 ACsÔÇŁ as a decisive threshold is **not validated**.

On threshold effects vs continuous increase: the evidence indicates both. The failure study finds a **clear interaction-round inflection** (diminishing returns after ~25 rounds). ţłÇciteţłéturn4view1ţłü SWE-PolyBench suggests a **scope inflection** around ÔÇť3+ filesÔÇŁ where success rates drop sharply in their setup. ţłÇciteţłéturn5view0ţłü At the same time, both studies also describe a general monotonic decline as complexity increases, implying there is a continuous component. ţłÇciteţłéturn4view0ţłéturn5view3ţłü Practically, this argues for a hybrid policy: treat some signals as ÔÇťhard triggersÔÇŁ (e.g., predicted multi-file cross-cutting changes; stalled >25 rounds) while maintaining a continuous complexity score for nuanced cases.

Given these findings, the most evidence-aligned conclusion is: **keep file scope in the policy, but lower its effective trigger (or weight it more heavily), and supplement it with signals that directly represent localization difficulty, verification complexity, and cross-cutting risk.** ţłÇciteţłéturn5view0ţłéturn4view1ţłéturn3view3ţłü

## Updated decomposition guidance

A decomposition policy that is consistent with the evidence should do two things your current rule cannot: (1) treat multi-file scope as a strong signal *but not the only one*, and (2) dynamically detect ÔÇťthis is going nowhereÔÇŁ during execution, because long-run looping is itself a dominant failure pattern. ţłÇciteţłéturn5view0ţłéturn4view1ţłü
